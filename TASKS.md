# TASKS.md

이 문서는 현재 `ai-ott-web` 프로젝트를 Claude Code에서 이어서 개발할 때 사용할 **실행 체크리스트**다.

목표는 POC 성격의 OTT 웹앱을 **운영형 MVP+** 수준까지 끌어올리는 것이다.  
현재 프로젝트는 홈/검색/카테고리/상세/재생/내 목록 흐름의 골격이 있으며, 이후 작업은 **품질 보강 + API 정합성 + 운영성 강화** 순서로 진행한다.

---

## 0. 작업 원칙

- 항상 **작은 단위로 수정하고 바로 검증**한다.
- UI만 바꾸지 말고 **백엔드 API 계약과 실제 데이터 shape**를 함께 확인한다.
- mock 데이터로 얼버무리지 않는다. 실제 API가 없으면 `TODO(API)`를 남기고 프록시와 타입부터 맞춘다.
- 기존 파일을 무분별하게 재작성하지 말고, **현재 구조를 존중하면서 점진적으로 개선**한다.
- 서버 컴포넌트와 클라이언트 컴포넌트의 책임을 섞지 않는다.
- 스타일보다 **정합성, 내결함성, 접근성, 성능**을 우선한다.

---

## 1. 현재 우선순위

우선순위는 반드시 아래 순서를 따른다.

1. **실행 안정화**
2. **API 연동 정합성 확보**
3. **핵심 사용자 흐름 완성**
4. **운영/어드민 보강**
5. **품질/접근성/성능 최적화**
6. **출시 준비 산출물 정리**

## 2. 가장 먼저 해야 할 일 (P0)

### P0-1. 프로젝트 실행 확인
- [ ] `package.json` 기준 실행 스크립트 확인
- [ ] 로컬 개발 서버 실행 확인
- [ ] 환경 변수 누락 여부 확인
- [ ] API 프록시 라우트가 정상 동작하는지 확인
- [ ] 빌드(`next build`) 시 치명적 오류가 없는지 확인

완료 기준:
- 로컬에서 홈, 검색, 카테고리, 상세, 재생, 내 목록 페이지가 모두 라우팅된다.
- 서버가 꺼져 있어도 어떤 API가 실패하는지 명확히 보인다.
- 환경 변수가 누락되면 조용히 실패하지 않고 명시적 오류가 난다.

### P0-2. 환경 변수 정리
- [ ] `.env.example`에 실제 필요한 모든 변수 반영
- [ ] 변수명을 일관되게 정리 (`API_BASE_URL`, `NEXT_PUBLIC_*` 등)
- [ ] 서버 전용/클라이언트 노출용 변수 분리
- [ ] 잘못된 환경 설정에 대한 가드 추가

완료 기준:
- 새 개발자가 `.env.example`만 보고 앱을 띄울 수 있다.

### P0-3. API 클라이언트 정리
- [ ] 공통 fetch 래퍼 정리
- [ ] timeout / 에러 파싱 / 상태 코드 핸들링 통일
- [ ] 요청/응답 타입 정의 위치 정리
- [ ] API별 응답 shape를 문서화된 타입으로 통일

완료 기준:
- 모든 API 호출이 공통 유틸을 사용한다.
- 임의의 `fetch()` 남발이 없다.

## 3. 핵심 사용자 흐름 완성 (P1)

### P1-1. 홈 화면
- [ ] 홈 섹션 렌더링 안정화
- [ ] 섹션 타입별 UI 분기 정리 (hero, rail, grid 등)
- [ ] 섹션 데이터 없을 때 graceful fallback 제공
- [ ] 포스터/배너 이미지 실패 시 대체 UI 제공
- [ ] 콘텐츠 카드 hover/focus 상태 개선

완료 기준:
- 홈이 빈 데이터, 일부 누락 데이터에서도 깨지지 않는다.

### P1-2. 검색
- [ ] 검색 입력 debounce 적용
- [ ] query string 기반 상태 동기화
- [ ] 검색 결과 없음 UI 제공
- [ ] 로딩/에러/빈 결과 상태 분리
- [ ] 엔터, ESC, 키보드 포커스 흐름 점검

완료 기준:
- 검색 URL 공유가 가능하고, 새로고침 시 상태가 유지된다.

### P1-3. 카테고리 페이지
- [ ] 카테고리 목록/상세 페이지 데이터 정합성 확인
- [ ] slug 없거나 잘못된 경우 404 처리
- [ ] 정렬/필터 UI 추가 가능한 구조로 정리
- [ ] 페이지네이션 또는 infinite scroll 도입 여부 결정

완료 기준:
- 카테고리 slug가 잘못되면 조용히 실패하지 않는다.

### P1-4. 콘텐츠 상세
- [ ] 상세 메타데이터 표시 정리
- [ ] 재생 CTA, 찜 CTA, 카테고리/태그 링크 정리
- [ ] related content 섹션 자리 마련
- [ ] 긴 설명 접기/펼치기 UX
- [ ] age rating/runtime/release year 표시 규칙 통일

완료 기준:
- 사용자가 상세 페이지에서 재생 또는 탐색으로 자연스럽게 이동할 수 있다.

### P1-5. 재생 페이지
- [ ] playback progress 저장 호출 안정화
- [ ] 재생 실패 상태 UI 추가
- [ ] 재생 가능 여부/권한 오류 대응 UI 마련
- [ ] 페이지 이탈 시 저장 처리 검토
- [ ] 추후 HLS/DRM 플레이어로 교체 가능한 구조로 정리

완료 기준:
- 재생 상태 저장이 과도하게 호출되지 않고, 실패해도 페이지가 깨지지 않는다.

### P1-6. 내 목록
- [ ] watchlist 페이지 empty state 개선
- [ ] 카드 제거/추가 후 optimistic UI 검토
- [ ] continue watching 섹션 구분
- [ ] 로그인/비로그인 상태 분기 고려

완료 기준:
- 내 목록이 단순 목록이 아니라 OTT 사용 흐름에 맞는 개인화 영역으로 보인다.

## 6. 백엔드와 함께 맞춰야 하는 항목 (중요)

아래는 웹만 수정해서 끝나지 않는다. 서버와 계약을 같이 맞춰야 한다.

### API 계약 확인 필요
- [ ] `/api/app/home`
- [ ] `/api/app/categories`
- [ ] `/api/app/categories/{slug}`
- [ ] `/api/app/catalog/search`
- [ ] `/api/app/catalog/contents/{id or slug}`
- [ ] `/api/app/me/watchlist`
- [ ] `/api/app/me/continue-watching`
- [ ] `/api/app/me/playback-progress/{contentId}`

각 API마다 확인할 것:
- [ ] 요청/응답 DTO 이름
- [ ] 필수/선택 필드

- [ ] 페이지네이션 구조
- [ ] 인증 필요 여부
- [ ] 실패 status code

완료 기준:
- 프론트 코드에서 추측으로 쓰는 필드가 없다.

## 11. 최종 목표 정의

이 프로젝트의 단기 목표는 다음이다.

- 홈/검색/카테고리/상세/재생/내 목록이 **깨지지 않고 일관되게 동작**할 것
- 운영자가 홈 편성과 콘텐츠 메타데이터를 **웹에서 수정 가능**할 것
- API 실패, 빈 데이터, 잘못된 slug, 이미지 누락 등 **현실적인 실패 상황을 견디는 UI**일 것
- 나중에 인증, 추천, DRM, OpenSearch, RBAC를 붙여도 **구조가 무너지지 않을 것**

이 문서는 진행하면서 계속 갱신한다.
